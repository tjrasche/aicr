# Dynamic Install-Time Values Demo

A deployment bundle is built once and often pulled many times, especially when it is pushed as a **pre-built OCI artifact**. By default every value is frozen into each component's `values.yaml` at bundle time, so a consumer who needs a different driver version, cluster name, or endpoint has to re-bundle.

Dynamic install-time values fix that. Declaring a value path as `dynamic` pulls it **out** of the baked `values.yaml` and into a separate `cluster-values.yaml`, which the installer consumes last. The artifact stays immutable and signable; the value stays settable at install time. This walkthrough verifies the value is genuinely overridable, not merely renamed.

> **`dynamic` vs `set`.** Use `dynamic` for values that genuinely vary per cluster (cluster name, subnet ID, region, endpoint URL). For a value that is static for this bundle but differs from the recipe default (e.g. a specific driver version), use `set` instead. This walkthrough exposes `gpuoperator:driver.version` purely because it is a simple, always-present path for demonstrating the mechanics â€” in practice a version pin is a `set` case.

The walkthrough is driven by a single `AICRConfig` passed with `--config`, so the exact same file works for an end user deploying and for testing this surface. The automated equivalent lives in the chainsaw suite [`tests/chainsaw/cli/bundle-dynamic/`](../tests/chainsaw/cli/bundle-dynamic).

This walkthrough covers:

1. Write the config.
2. Generate a recipe (`aicr recipe --config`).
3. Build the bundle (`aicr bundle --config`).
4. Inspect the layout (`cluster-values.yaml` + `install.sh`).
5. Confirm the value actually moved.
6. Override the value at install time, no re-bundle.
7. Push as a pre-built OCI artifact and confirm the split survives.
8. Deployer differences (helm / helmfile / argocd-helm / flux).

## Prerequisites

* `aicr` on `$PATH` (any build: this flow is offline, no signing or cluster).
* For step 7 only: [`oras`](https://oras.land/) to pull the artifact back, and outbound network to [`ttl.sh`](https://ttl.sh) (an anonymous, ephemeral OCI registry: no setup, no auth, images auto-expire). No local registry needed.

## 1. Write the config

One `AICRConfig` is the single source of truth for the recipe criteria and the bundle inputs. Drop this into `aicr-config.yaml`:

```shell
cat > aicr-config.yaml <<'EOF'
kind: AICRConfig
apiVersion: aicr.run/v1alpha2
metadata:
  name: dynamic-values-demo
spec:
  recipe:
    criteria:
      service: eks
      accelerator: gb200
      os: ubuntu
      intent: training
    output:
      path: recipe.yaml
  bundle:
    input:
      recipe: recipe.yaml
    output:
      target: ./bundle
    deployment:
      deployer: helm
      dynamic:
        - gpuoperator:driver.version    # component:path â€” exposed at install time
EOF
```

The `dynamic` list takes `component:path` entries (the `component` is the value-override key, the same one `set` uses). Add more entries to expose more paths. CLI flags always win over the matching config field, which step 7 uses to retarget the output without editing the file.

## 2. Generate the recipe

```shell
aicr recipe --config aicr-config.yaml
```

Writes `recipe.yaml` per `spec.recipe.output.path` (14 components for this criteria).

## 3. Build the bundle

```shell
aicr bundle --config aicr-config.yaml
```

Writes `./bundle/` per `spec.bundle.output.target`, with `driver.version` split out into `cluster-values.yaml` because it is declared `dynamic`.

> To see the contrast, comment out the `dynamic:` block and re-bundle: with no dynamic paths, `driver.version` stays baked in `values.yaml` and `cluster-values.yaml` is emitted empty.

The exposed value now lives in `cluster-values.yaml`:

```shell
cat ./bundle/010-gpu-operator/cluster-values.yaml
# # Generated by AICR
# ---
# driver:
#   version: 580.126.20
```

And `install.sh` applies it after the baked values, so it wins:

```shell
grep -n 'cluster-values.yaml' ./bundle/010-gpu-operator/install.sh
# 39:  -f values.yaml -f cluster-values.yaml \
```

`-f values.yaml -f cluster-values.yaml` is ordered: Helm's last `-f` wins, so whatever sits in `cluster-values.yaml` overrides the baked default.

## 5. Confirm the value moved

Three quick checks, the same ones the chainsaw suite enforces:

```shell
# a) gone from the baked values.yaml (no match)
grep -c 'version: 580' ./bundle/010-gpu-operator/values.yaml
# 0

# b) present in cluster-values.yaml
grep version ./bundle/010-gpu-operator/cluster-values.yaml
#   version: 580.126.20

# c) install.sh references cluster-values.yaml
grep -q cluster-values.yaml ./bundle/010-gpu-operator/install.sh && echo wired
# wired
```

If all three hold, the value is exposed for install-time override.

## 6. Override at install time, no re-bundle

Edit `cluster-values.yaml` and the next install picks it up; the bundle itself is untouched:

```shell
cat > ./bundle/010-gpu-operator/cluster-values.yaml <<'EOF'
# Generated by AICR
---
driver:
  version: 580.140.99
EOF

# install.sh now deploys with driver.version=580.140.99
# (helm renders -f values.yaml -f cluster-values.yaml, last value wins)
```

To preview the override without a cluster, render with Helm directly:

```shell
helm template gpu-operator <chart> \
  -f ./bundle/010-gpu-operator/values.yaml \
  -f ./bundle/010-gpu-operator/cluster-values.yaml \
  | grep -A2 'driver'
# version: 580.140.99
```

**Success:** the rendered manifests carry the overridden value, produced from the same bundle with no re-bundle step.

## 7. Pre-built OCI artifact

The dynamic split happens during bundle generation, so it survives being pushed as an OCI artifact. Retarget the *same config* to an OCI ref with a `--output` override (CLI wins over `spec.bundle.output.target`), push to [`ttl.sh`](https://ttl.sh) (anonymous, ephemeral, no setup), pull back, and re-inspect. The tag is the time-to-live; use a unique image name since the namespace is shared and public:

```shell
REF="ttl.sh/aicr-bundle-$(uuidgen | tr '[:upper:]' '[:lower:]'):1h"

# same config, output overridden to the registry (ttl.sh is HTTPS, no --plain-http)
aicr bundle --config aicr-config.yaml --output "oci://$REF"

# pull the artifact back and confirm cluster-values.yaml is intact
oras pull "$REF" -o ./b-oci
cat ./b-oci/010-gpu-operator/cluster-values.yaml
#   version: 580.126.20
```

> ttl.sh is public; anything pushed is world-readable until it expires. Fine for a throwaway demo bundle, not for anything sensitive â€” push to your own registry by pointing `spec.bundle.output.target` (or `--output`) at it, e.g. `oci://registry.example.com/aicr-bundle:tag`. Transport is separate: `spec.bundle.registry.plainHTTP: true` only switches HTTP instead of HTTPS (enable it only when the registry is intentionally served over plain HTTP, such as a local test registry); leave it off for any HTTPS registry, and use `spec.bundle.registry.insecureTLS: true` to skip TLS verification against a self-signed HTTPS registry.

A consumer pulls the pre-built artifact, edits `cluster-values.yaml`, and installs, exactly as in step 6. Same immutable artifact, different install-time values.

## 8. Deployer differences

`dynamic` is supported by every deployer **except plain `argocd`**, and the install-time override surface differs. Set `spec.bundle.deployment.deployer` accordingly:

| Deployer | Where the dynamic value is overridden at install time |
|----------|-------------------------------------------------------|
| `helm` (default) / `helmfile` | Edit `cluster-values.yaml`; `install.sh` / helmfile applies it last (`-f values.yaml -f cluster-values.yaml`, last wins). |
| `argocd-helm` | `dynamic` pre-populates paths in the chart's root `values.yaml`; override at install via `helm install â€¦ --set <key>=<value>`. |
| `argocd` | **Not supported.** A `dynamic` declaration with `deployer: argocd` is rejected â€” use `argocd-helm` for install-time values. |
| `flux` | Dynamic paths go into a per-component `configmap-values.yaml`, referenced by the HelmRelease via `spec.valuesFrom` (merged before inline values). |

Pick the deployer to match how the consumer installs; the dynamic value lands in the right place for that path.

## Troubleshooting

**`cluster-values.yaml` is empty after a `dynamic` declaration** â€” the `component:path` did not resolve. Check the component prefix is its value-override key (e.g. `gpuoperator`, not `gpu-operator`) and that the path exists in the baked `values.yaml`.

**Override seems ignored at install** â€” confirm `install.sh` lists `-f cluster-values.yaml` *after* `-f values.yaml`. With Helm the last `-f` wins; the order matters.

**`dynamic` rejected** â€” you set `deployer: argocd`. Switch to `argocd-helm` (or `helm` / `helmfile` / `flux`), which support install-time values.
