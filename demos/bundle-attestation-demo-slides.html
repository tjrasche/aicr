01 · The problem

Generated artifacts have no upstream tag

A deployment bundle is built locally from a recipe — there’s no registry to anchor trust to. Two questions an operator still has to answer before applying it to a production cluster.

QuestionWhat proves it
Who built this bundle?Sigstore signature pinned to the creator’s OIDC identity
Which AICR CLI built it — and was that CLI itself attested?Bundle predicate references the binary’s SLSA attestation

Without attestation

checksums.txt detects accidental payload or tree changes, but provides no signed creator identity or provenance.

With --attest

The complete closed-world payload inventory, including recipe.yaml, is cryptographically bound to the creator and the binary that produced it. Recursive provenance.

aicr bundle --attest

producer
$ aicr bundle --recipe recipe.yaml --output ./my-bundle --attest
# In GitHub Actions: ambient OIDC is detected automatically.
# Locally: a browser opens for keyless OIDC sign-in.
03 · The chain

Bundle ↔ binary ↔ NVIDIA CI

Why both halves matter The bundle signature alone proves a creator made a bundle. The chain to the binary attestation proves the bundle was built by an attested NVIDIA-CI-released AICR CLI — not by a fork, an alpha build, or a tampered binary. AICR generates manifest entries sorted, but validates valid entries in any order; reordering a signed manifest changes its signed bytes and invalidates this chain.
04 · Verify

aicr verify — clean pass

happy path
$ aicr verify ./my-bundle

✓ Checksums verified (12 files)
✓ Bundle attested by: jdoe@company.com
✓ Binary built by: https://github.com/NVIDIA/aicr/.github/workflows/on-tag.yaml@refs/tags/v1.0.0
✓ Identity pinned to NVIDIA CI
  Trust level: verified

Bundle verification: PASSED

Five gates ran: closed-world inventory, bundle signature, bundle predicate, binary attestation chain, identity pin. Additional files or directories, symlinks, other non-regular objects, and failed attestations report unknown.

05 · Policy

Four trust levels

LevelNameCriteria
4verifiedChecksums  +  bundle attestation  +  binary attestation pinned to NVIDIA CI
3attestedBundle attestation verified; binary attestation missing, or external data used
2unverifiedClosed-world checksum inventory valid, no --attest was passed
1unknownMissing, invalid, or incomplete manifest; unexpected filesystem entry; or failed attestation
Why levels, not pass/fail Different bundles live under different threats. CI-built production bundles must be verified; an on-call engineer’s emergency hotfix bundle might only reach attested. Legacy bundles with incomplete manifests remain unknown and must be regenerated. Operators pick the floor per environment.
06 · Enforce

Policy from one flag

policy flags
# Floor on trust level — fail any bundle below the threshold.
$ aicr verify ./my-bundle --min-trust-level verified

# Require a specific creator (or set of identities).
$ aicr verify ./my-bundle --require-creator jdoe@company.com

# Constrain the CLI version that produced the bundle.
$ aicr verify ./my-bundle --cli-version-constraint ">= 1.0.0"

# CI: keep the verify exit code; failed integrity or attestation reports unknown
$ set -o pipefail; aicr verify ./my-bundle --format json | jq '.trustLevel, .trustReason'

--min-trust-level verified

Production gate. Demands the full chain back to NVIDIA CI.

--min-trust-level attested

Pre-prod gate. Allows external data sources while still requiring a signature.

06 · Enforce

Tamper-evident

tampered bundle
$ f=$(ls my-bundle/[0-9]*-*/values.yaml | head -1)
$ echo '# tampered' >> "$f"
$ aicr verify ./my-bundle

✗ Checksums failed: 1 file mismatch
    <NNN-component>/values.yaml — sha256 mismatch (got 3f9a…, want 7b21…)
  Trust level: unknown

Bundle verification: FAILED (non-zero exit)

The signature’s subject is the digest of checksums.txt, and checksums.txt pins every regular payload file. Mutating a listed file breaks the checksum; mutating or reordering checksums.txt breaks the signature. Adding any unlisted file, directory, symlink, or other non-regular object fails exact-tree verification.

07 · Where it fits

One trust model, three subjects

Images & binaries

SLSA provenance + signed SPDX SBOM on every published image (CLI binaries ship a separate, unsigned SBOM asset), signed by NVIDIA CI. demos/provenance.md

Deployment bundles

aicr bundle --attest signs the generated bundle with build provenance + trust levels. You are here.

Recipe evidence

A signer-bound validation result for a recipe (proves who signed it, not cluster physicality), OCI-distributed with an in-tree pointer. demos/evidence.md

Same primitives

Sigstore keyless + Fulcio + Rekor + in-toto across the board — different subjects, one verification model.

Summary aicr bundle --attest → aicr verify --min-trust-level verified. Two commands, signed proof of who built it, which CLI built it, and an enforceable policy boundary at apply time.
